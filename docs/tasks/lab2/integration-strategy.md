# Стратегия интеграционного тестирования (Bottom-Up)

## Цель
Построить интеграционные тесты снизу вверх: от атомарных модулей с моками до полной системы на реальных вычислениях.

## Этапы
1. Листовые модули с моками зависимостей.
Проверяем, что модуль:
- вызывает нужные зависимости с корректными аргументами;
- корректно обрабатывает ветки с исключениями.

Покрытие:
- `Cos <- sin`
- `Tan <- sin, cos`
- `Cot <- sin, cos`
- `Sec <- cos`
- `Csc <- sin`
- `Logarithm <- ln`

2. Подсистемы с моками.
Проверяем сборку формулы и взаимодействие модулей без шума от рядов.

Покрытие:
- `LogSystem <- log3, log10`
- `TrigSystem <- sec, cos, csc, tan, cot`

3. Маршрутизация системы с моками.
Проверяем, что `SystemFunction` корректно выбирает ветку:
- `x <= 0` -> trig;
- `x > 0` -> log.

4. Замена моков на реальные зависимости.
Проверяем интеграцию через реальные `SeriesSin` и `SeriesLn`:
- тригонометрическая цепочка `sin -> cos/tan/cot/sec/csc -> TrigSystem`;
- логарифмическая цепочка `ln -> log3/log10 -> LogSystem`;
- полная система `TrigSystem + LogSystem -> SystemFunction`.

Существующие тесты этапа 4 сохранены и продолжают проверять работу на реальных значениях.

## Критерии готовности
- Все тесты с моками зелёные: взаимодействия и исключения подтверждены.
- Все интеграционные тесты на реальных зависимостях зелёные.
- Ветка `x <= 0` стабильно даёт ожидаемую ошибку для текущей формулы trig-части.
- Ветка `x > 0` совпадает с эталонными значениями в допусках.
