# Отчёт по лабораторной работе №3

## Текст задания

Вариант №36958: `Tutu.Ru` (`https://www.tutu.ru/`).

Необходимо:

- сформировать варианты использования сайта;
- разработать на их основе тестовое покрытие;
- провести функциональное тестирование интерфейса сайта;
- автоматизировать тестирование средствами Selenium;
- использовать XPath-селекторы для работы с динамически формируемыми элементами.

Примечание по реализации: вместо устаревшего Selenium RC используется `Selenium WebDriver 4`. Шаблон Selenium IDE приложен в формате `.side`.

## Детализированные прецеденты

### UC-01. Построить маршрут (откуда/куда/дата)

- Идентификатор: `UC-01`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Заинтересованные стороны: пользователь (быстрый подбор маршрута), сервис (корректные параметры поиска)
- Предусловия: открыта главная страница, блок поиска маршрута доступен
- Триггер: пользователь хочет найти варианты поездки
- Основной успешный сценарий:
1. Пользователь вводит пункт отправления.
2. Пользователь вводит пункт назначения.
3. Пользователь выбирает дату поездки.
4. Пользователь запускает поиск.
5. Система открывает страницу результатов с переданными параметрами.
- Альтернативные потоки:
1. A1: введён несуществующий населённый пункт; система показывает подсказку и не даёт продолжить до корректного значения.
2. A2: дата в прошлом; система отображает ошибку валидации даты.
- Постусловия:
1. Успех: открыта страница результатов поиска.
2. Минимальные гарантии: введённые значения сохранены в форме.
- Спецтребования: отклик автодополнения не более 2 секунд.

### UC-02. Изменить параметры маршрута и дату

- Идентификатор: `UC-02`
- Уровень: расширение `UC-01`
- Основной актор: гость сайта
- Предусловия: пользователь находится на форме поиска с заполненными полями
- Триггер: пользователь хочет скорректировать маршрут
- Основной успешный сценарий:
1. Пользователь меняет направление (или использует кнопку обмена "откуда/куда").
2. Пользователь меняет дату поездки.
3. Пользователь повторно запускает поиск.
4. Система обновляет выдачу по новым параметрам.
- Альтернативные потоки:
1. A1: пользователь оставил обязательное поле пустым; система подсвечивает поле и блокирует поиск.
2. A2: пользователь выбрал недопустимую комбинацию дат; система показывает диагностическое сообщение.
- Постусловия:
1. Успех: выдача соответствует обновлённым параметрам.
2. Минимальные гарантии: пользователь остаётся на странице с подсказкой по ошибке.
- Спецтребования: изменение параметров не должно приводить к сбросу остальных полей без явного действия пользователя.

### UC-03. Переключиться между сервисами в шапке

- Идентификатор: `UC-03`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Предусловия: открыта главная страница
- Триггер: пользователь выбирает другой сервис (авиа, ж/д, автобус, отели, электрички)
- Основной успешный сценарий:
1. Пользователь нажимает на сервис в шапке.
2. Система переключает активный раздел.
3. Система отображает соответствующую форму поиска.
4. URL и заголовок отражают выбранный сервис.
- Альтернативные потоки:
1. A1: сервис временно недоступен; система показывает уведомление и сохраняет текущий раздел.
- Постусловия:
1. Успех: активен выбранный сервис и его форма.
2. Минимальные гарантии: пользователь остаётся на доступной странице.
- Спецтребования: визуальная индикация активного таба обязательна.

### UC-04. Перейти в справочную и открыть статью

- Идентификатор: `UC-04`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Предусловия: открыта страница с доступной ссылкой `Справочная`
- Триггер: пользователь ищет инструкцию
- Основной успешный сценарий:
1. Пользователь кликает ссылку `Справочная`.
2. Система открывает раздел `/2read/`.
3. Пользователь выбирает категорию.
4. Пользователь открывает статью.
- Альтернативные потоки:
1. A1: статья не найдена по категории; система предлагает соседние категории и поиск по справке.
- Постусловия:
1. Успех: открыта статья из справочного раздела.
2. Минимальные гарантии: открыт раздел справки.
- Спецтребования: ссылка в шапке должна быть доступна с каждой ключевой страницы.

### UC-05. Заполнить форму поиска отелей

- Идентификатор: `UC-05`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Предусловия: активирован сервис `Отели`
- Триггер: пользователь планирует проживание
- Основной успешный сценарий:
1. Пользователь вводит город или объект размещения.
2. Пользователь выбирает даты заезда/выезда.
3. Пользователь задаёт число гостей.
4. Пользователь запускает поиск.
5. Система открывает результаты для отелей.
- Альтернативные потоки:
1. A1: город не выбран из подсказки; система просит уточнить значение.
2. A2: дата выезда раньше даты заезда; система показывает ошибку в форме.
- Постусловия:
1. Успех: открыта выдача отелей.
2. Минимальные гарантии: поля формы содержат последние валидные значения.
- Спецтребования: календарь должен блокировать выбор прошедших дат.

### UC-06. Подписаться на email-рассылку

- Идентификатор: `UC-06`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Предусловия: блок подписки доступен на странице
- Триггер: пользователь хочет получать рассылку
- Основной успешный сценарий:
1. Пользователь вводит валидный email.
2. Пользователь отмечает чекбокс согласия на обработку данных.
3. Пользователь нажимает `Подписаться`.
4. Система подтверждает успешную отправку формы.
- Альтернативные потоки:
1. A1: не отмечено согласие; система не отправляет форму и показывает причину.
- Постусловия:
1. Успех: подписка создана.
2. Минимальные гарантии: введённый email остаётся в поле.
- Спецтребования: согласие обязательно до отправки формы.

### UC-07. Обработать ошибки формы подписки

- Идентификатор: `UC-07`
- Уровень: расширение `UC-06`
- Основной актор: гость сайта
- Предусловия: открыт блок подписки
- Триггер: пользователь вводит некорректный email или пропускает обязательные поля
- Основной успешный сценарий:
1. Пользователь вводит email в неверном формате.
2. Пользователь нажимает `Подписаться`.
3. Система валидирует форму.
4. Система показывает сообщение об ошибке и не отправляет данные.
- Альтернативные потоки:
1. A1: email корректен, но чекбокс согласия не выбран; система показывает отдельную ошибку по согласию.
- Постусловия:
1. Успех: некорректные данные не отправлены, пользователь получил обратную связь.
2. Минимальные гарантии: форма остаётся доступной для исправления.
- Спецтребования: ошибки валидации должны быть привязаны к конкретным полям.

### UC-08. Вернуться на главную после навигации

- Идентификатор: `UC-08`
- Уровень: пользовательская цель
- Основной актор: гость сайта
- Предусловия: пользователь находится в дочернем разделе (сервис или справка)
- Триггер: пользователь хочет вернуться к стартовой странице
- Основной успешный сценарий:
1. Пользователь использует логотип или пункт `Главная`.
2. Система открывает главную страницу.
3. Система отображает основной набор сервисов и ключевые блоки.
- Альтернативные потоки:
1. A1: переход выполняется в новой вкладке; система сохраняет состояние предыдущей вкладки.
- Постусловия:
1. Успех: пользователь возвращён на главную страницу.
2. Минимальные гарантии: есть доступ к основной навигации.
- Спецтребования: переход должен выполняться не более чем за 3 секунды в типовом окружении.
